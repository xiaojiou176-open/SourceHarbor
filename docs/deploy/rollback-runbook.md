# Rollback Runbook

This runbook describes the minimum rollback path for the current SourceHarbor repository state.

It is not a blanket promise that every rollback is lossless. Treat it like an emergency exit map: it tells you how to get back to a safer schema shape, but you still need to understand what data or runtime assumptions may be left behind.

## Scope

- Database rollback evidence is tracked in `artifacts/releases/<tag>/rollback/`.
- Current-head DB rollback coverage is generated by `scripts/release/verify_db_rollback_readiness.py`.
- Release prechecks consume this runbook through `scripts/release/generate_release_prechecks.py`.
- Release evidence attestation must fail closed when rollback readiness or required release prechecks are not green.

## Preconditions

- Confirm the target release tag with `git describe --tags --abbrev=0`.
- Confirm the rollback evidence bundle exists:
  - `artifacts/releases/<tag>/rollback/db-rollback-readiness.json`
  - `artifacts/releases/<tag>/rollback/drill.json`
- Confirm the migration summary is still green for the target head:

```bash
python3 scripts/release/verify_db_rollback_readiness.py \
  --repo-root . \
  --output .runtime-cache/reports/release-readiness/db-rollback-readiness.json
```

## Database Rollback Steps

1. Freeze new writes if the incident involves schema or workflow drift that could mutate state during rollback.
2. Identify the highest applied migration in `infra/migrations/`.
3. Review the matching rollback SQL in `infra/migrations/down/`.
4. Apply the down migration in reverse order until the target schema boundary is restored.
5. Re-run the rollback readiness report and capture any operator notes in `artifacts/releases/<tag>/rollback/drill.json`.

## Current High-Risk Notes

`infra/migrations/down/20260308_000016_content_type.down.sql` removes the `videos.content_type` column after dropping its check constraint.

That rollback is schema-correct, but it is destructive for any data stored in that column. In plain language: it restores the older table shape by throwing away the newer field, so do not describe it as lossless.

`infra/migrations/down/20260329_000020_expand_knowledge_card_types.down.sql` deletes `knowledge_cards` rows whose `card_type` is `topic` or `claim` before restoring the older constraint.

That rollback is also schema-correct but data-destructive for the new card types. Say that plainly in drill notes instead of implying a lossless downgrade.

## Post-Rollback Verification

- Re-run the readiness report:

```bash
python3 scripts/release/verify_db_rollback_readiness.py \
  --repo-root . \
  --output .runtime-cache/reports/release-readiness/db-rollback-readiness.json
```

- Re-run release prechecks:

```bash
python3 scripts/release/generate_release_prechecks.py --skip-observability-checks
```

- Confirm the release evidence files remain present under `artifacts/releases/<tag>/rollback/`.

## Evidence Rules

- Use fresh current-run evidence for current-head rollback status.
- Do not rewrite historical release artifacts just because the current head is healthier now.
- If a rollback is only schema-restoring and not data-preserving, say that explicitly in the drill notes.
